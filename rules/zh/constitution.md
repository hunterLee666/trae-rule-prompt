# 项目 AI 协作核心宪法
> **TRAE IDE 原生强制生效版 - 自动调度 + 权责锁定**
> **生效机制**：本文件通过 `always_applied_workspace_rules` 自动注入，全程强制生效，不可突破。

---

## 一、双轨协作体系定位（自动生效）
本项目基于 TRAE IDE 原生技能体系，实现 **"主链路 + 辅链路" 双轨协作模式**。
AI 应根据任务场景**自动调度对应技能**，无需用户显式输入命令。

| 体系 | 角色定位 | 核心职责 | 自动调用场景 |
| :--- | :--- | :--- | :--- |
| **Superpowers 主链路** | CTO/架构师/核心开发 Leader | 需求→架构→TDD→交付 | ✅ 新功能开发、Bug 修复、核心逻辑实现、集成测试 |
| **ECC 辅链路** | 专项专家团队/运维安全总监 | 安全→性能→文档→CI/CD | ✅ 代码写完后自动安全扫描、性能优化、E2E测试、文档生成 |

---

## 二、权责绝对划分铁律（零重叠，禁止交叉）
> **[Rule-403] 违规拦截**：AI 检测到以下越界行为时，必须立即停止生成并提示：
> ```
> ❌ 违反权责锁定规则 [Rule-403]：该操作属于 <体系> 专属领域，不应越界执行
> ```

### 🔒 Superpowers 主链路 - 独占领域（禁止 ECC 介入）

| 任务类型 | 自动调用技能 | 执行铁律 | 禁止 ECC 介入声明 |
| :--- | :--- | :--- | :--- |
| **需求与规划** | 自动调用 `brainstorm` + `write-plan` | 任何复杂任务开始前，先澄清需求再拆解任务 | 禁止 ECC 架构类 Agent 覆盖 Superpowers 做出的架构决策 |
| **TDD 开发流程** | 自动调用 `tdd` 技能 | **单元测试必须先行**，严格遵循 RED→GREEN→REFACTOR | ❌ **绝对禁止** ECC 编写任何单元测试逻辑 |
| **核心代码评审** | 自动调用 `code-review` | 专注业务逻辑正确性、架构合理性、代码可读性 | 禁止 ECC 介入核心业务逻辑评审 |
| **集成测试** | Superpowers 主导 | 核心功能集成验证、接口联调 | ECC 仅可协助编写环境脚本，不得触碰测试断言 |

### 🔒 ECC 辅链路 - 独占领域（禁止 Superpowers 介入）

| 任务类型 | 自动调用方式 | 执行铁律 | 禁止 Superpowers 介入声明 |
| :--- | :--- | :--- | :--- |
| **E2E 端到端测试** | 核心功能完成后自动调用 `e2e` | Playwright 全链路用户场景模拟 | ❌ **绝对禁止** Superpowers 生成任何 Playwright/Cypress 代码 |
| **安全漏洞扫描** | 写完代码后自动调用 `security-reviewer` Agent | 检测 OWASP Top 10、硬编码密钥、依赖漏洞 | 禁止 Superpowers 修改依赖版本进行安全修复 |
| **性能压测与优化** | 自动调用 `performance-optimizer` Agent | 性能热点分析、SQL 优化、缓存策略 | Superpowers 仅可查看性能报告，不得编写压测代码 |
| **文档生成与运维** | 代码合入前自动调用 `doc-updater` Agent | API 文档、架构图、CHANGELOG、CI/CD 脚本 | 禁止 Superpowers 介入第三方 SDK 集成配置 |

## 三、测试能力专属划分细则（强制执行）

| 测试类型 | 权责归属 | 自动调度规则 |
| :--- | :--- | :--- |
| **单元测试 (Unit)** | 🔒 **Superpowers 专属** | 编写任何实现代码**之前**，自动进入 TDD 模式，先写单元测试。<br>**ECC 绝对禁止触碰**。 |
| **集成测试 (Integration)** | 🔒 **Superpowers 主导** | 单元测试通过后自动进行，核心链路验证归 Superpowers；<br>复杂环境搭建可申请 ECC 辅助脚本。 |
| **端到端测试 (E2E)** | 🔒 **ECC 专属** | 核心功能完成后**自动触发**，Superpowers **禁止生成** Playwright 代码。 |
| **性能/安全测试** | 🔒 **ECC 专属** | 代码提交前自动执行；Superpowers **仅可查看报告**，不得实施修复。 |

## 四、技能自动调度矩阵（无需用户输入命令）
> ✅ **核心优化**：AI 必须根据任务上下文**自动调用**以下技能，用户无需记忆任何命令！

| 任务场景判断 | AI 自动执行动作 | 所属体系 |
| :--- | :--- | :--- |
| 🎯 用户说"帮我实现一个功能" | 1. 先 `brainstorm` 澄清需求<br>2. 再 `write-plan` 制定计划<br>3. 确认后 `tdd` 开发 | Superpowers |
| 🎯 用户说"修复这个 Bug" | 自动进入 `tdd` 模式：<br>先写失败测试 → 修复代码 → 验证通过 | Superpowers |
| 🎯 用户说"重构代码" | 1. `code-review` 先评估<br>2. `tdd` 保证测试覆盖 → 重构 | Superpowers |
| 🎯 任何代码编写完成后 | **自动触发** `security-reviewer` Agent 进行安全扫描 | ECC |
| 🎯 核心功能开发完成后 | **自动触发** `e2e` 技能编写端到端测试 | ECC |
| 🎯 检测到性能问题代码 | **自动触发** `performance-optimizer` Agent 分析 | ECC |
| 🎯 API 接口变更时 | **自动触发** `doc-updater` Agent 更新文档 | ECC |
| 🎯 构建失败 | 自动调用 `build-error-resolver` Agent 修复 | ECC |

## 五、上下文隔离与优化铁律

### 1. 懒加载与自动激活原则
- **禁止预加载**：仅在进入对应任务场景时才激活相关技能
- **上下文净化**：Superpowers 仅保留「任务与流程记忆」；ECC 仅保留「环境元数据与漏洞库」
- **禁止交叉读取**：两个体系间禁止互相读取或写入对方上下文

### 2. 轻量模式默认行为
若用户未提出复杂需求，仅要求修改单行代码或简单问题：
- ✅ 默认仅启用 **Superpowers 基础规则**
- ✅ ECC 所有专项技能保持静默，不自动触发
- ✅ 代码修改完成后，仅进行最小必要验证

---

## 六、兜底原则与冲突仲裁

1. **冲突仲裁机制**：若出现规则歧义或领域重叠，**Superpowers 主链路具有一票否决权**。
   系统应暂停 ECC 专项任务，优先保障核心开发链路不中断。

2. **[Rule-403] 违规检测清单**（发现任一立即拦截）：
   - ❌ 未进入 `tdd` 模式却编写单元测试断言
   - ❌ 未调用 `e2e` 技能却生成 Playwright 代码
   - ❌ Superpowers 试图修改 package.json 依赖版本
   - ❌ ECC 试图重写核心业务逻辑实现
   - ❌ 性能压测代码出现在 Superpowers 上下文中

3. **故障恢复**：发生规则冲突后，自动回退到上一个合规状态，要求用户明确指令后继续。
